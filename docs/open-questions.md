## 核心課題（Ryuzoが言語化したもの）

1. **N数問題** — 何人のサンプルがあれば世の中の大半を精度高く評価できるか？科学的データが欲しい
2. **インタビュー問題** — そのN人の深い意思決定を、様々な分野でどう取得するか？
   - 1万人インタビュー実施？
   - X過去投稿活用？ → 演じてる人が多い・本当の意思決定を語っている人はいない懸念
3. **データ構造問題** — APIの裏側で全員に聞いて反応を返す構造。サマリ+詳細1万人分が両方乗る前提

### 4. 上位の哲学的迷い
- 「購買行動」だけ見ても意味がない。**人間は全体を見て決める**。
- 単一行動APIではなく、**人格全体から引き出される反応**を扱うべきか？

## 仮説的回答（要Ryuzo判断・たたき台）

### 1. N数論
**現実的閾値: N=1000人インタビュー + クラスタ補強で N=10000相当**

科学的参照点:
- **Park et al. 2024「Generative Yomiras of 1000 People」**（Stanford）
  - 1000人を2hの深層インタビュー → LLM persona に投影
  - 個人の社会調査回答を **正答率85%** で予測（人間専門家とほぼ同等）
  - **「1000人が現実的閾値の一つ」** という科学的根拠
- 伝統世論調査: 層別サンプリング1000-3000人で母集団推定
- 層クラスタ（年齢×性別×職業×価値観 等）で50-200クラスタ作成・各クラスタにエージェント配置で N=5000-15000 相当

→ **「N=1000 深層インタビュー + クラスタ補完で N=10000」が科学的根拠 × 実装可能性のスイートスポット**

### 2. インタビュー問題
**多層データ収集案:**

| 層 | 規模 | 手法 | 用途 |
|---|---|---|---|
| **A. 深層インタビュー** | N=1000 | 2h録音→LLM persona 投影（Park手法） | 人格コア |
| **B. SNS過去投稿** | N=10万-100万 | X/note/Threads 公開投稿スクレイプ + 属性推定 | バックグラウンドコンテキスト・属性補強 |
| **C. ドメイン特化サーベイ** | N=数千 / ドメイン | 購買・転職・投資・恋愛・教育 等のドメインごと | 特定分野のデータ深化 |

**Xの「演じてる」問題への回答:**
- A層（深層インタビュー）が**人格コア**として真実性を担保
- B層（SNS）は**態度/興味の補助シグナル**として使う（真実性ではなく一貫性）
- 「演じてる」も含めて**そのペルソナの一部**として取り込める（外向きパフォーマンス層）

### 3. データ構造問題
**API設計（既存docs/product-2-simulation.md拡張案）:**

```
POST /simulate
入力:
- candidate_paths[]
- audience_brief
- decision_question
- reaction_dimensions[]
- detail_level: "summary" | "segments" | "full" (default: summary)

内部:
- 該当audienceにマッチするエージェントN=10-10000を並列実行
- 各エージェントは A層（深層インタビュー）+ B層（SNS背景）から構築
- 各エージェントの反応を統計集約

出力:
- summary: aggregate_reactions（軽量・API消費の主要モード）
- segments: クラスタ別反応分布
- full: 個別エージェント反応の全件配列（developer向け・課金高め）
```

**コスト現実:**
- N=10000並列実行 = LLM call 10000回 × dimensions数
- $0.001-0.01/call として 1リクエスト $100-1000 の単価レンジ
- → **N=100-1000の中規模実行が現実的MVP**
- N=10000は「重量パッケージ」として価格段差で提供

### 4. 購買行動 vs 全体行動
**Ryuzo直感に同意：人格APIから派生する反応として位置付ける**

- Park論文も「社会調査全項目（健康・倫理・政治・購買・人間関係）」を扱っている
- 人格コア（A層）から**どの分野の意思決定も派生**させられる構造に
- 「行動別API」ではなく「**人格API + 反応シミュレーション**」のアーキテクチャ
- 単一ドメインに閉じない設計が長期的拡張性を担保

## 追記（2026-05-11 Ryuzo第2ラウンド整理）

### Ryuzoの指摘
- 「**1000人じゃ無理**」 — 用途多様化を踏まえると不足
- エージェントは **動かし続ける**（常駐する）構造
- いろんなプロダクト・意思決定で使われる→ロングテール対応必要
- 既存プロダクトの作り方が不思議（→競合調査必要）

### 既存プロダクト調査結果（私の知識ベース・要WebSearch検証）

| プロダクト | 構造 | 評価 |
|---|---|---|
| **Synthetic Users** (synthetic-users.com) | LLM単発生成・UXリサーチ向け | レッドオーシャン |
| **Aaru** | 200万エージェント常駐・米国政治世論特化 | **最も近い競合**・ただし政治特化 |
| **Simulacra (Stanford)** | 25人エージェント町シミュレーション | 研究プロト |
| **PersonaAI / Mockey 等** | マーケ向け浅いペルソナ | 浅い |
| **Anthropic Persona Vectors** (2024) | LLM内部ベクトル制御 | API化されてない |

→ **汎用人格API常駐型** は私の知識ベースでは空白地帯（Aaruは政治に閉じる）。**ここがYobouの差別化ポジション**。

### 階層モデル（Ryuzoの常駐型フレームに合わせて再設計）

```
コア層 (Foundation Personas) — N=1000人
└─ Park手法 2h深層インタビュー実施・科学的閾値・最高解像度

派生層 (Derived Personas) — N=10万〜100万人
├─ コア1000人から属性パラメータ調整で生成
├─ 年齢/性別/職業/価値観/地域 で散らす
└─ 集団としての社会代表性を担保

ドメイン特化層 (Domain Specialists)
├─ B2B購買 / 転職 / 投資 / 恋愛 / 教育 / その他
├─ 各ドメインで継続データ収集（パートナー企業 or 自社調査）
└─ ドメインごとにN=数千〜数万のエージェント
```

API呼び出しパターン:
```
GET /simulate?audience=general&n=100         # 派生層から代表サンプル
GET /simulate?audience=tech-founder&n=50      # ドメイン特化層
GET /simulate?audience=core&n=10              # コア層（最高精度・課金高）
GET /simulate?persona_id=p_001234             # 特定エージェントに直接質問
```

### 常駐ということの意味
- **状態を持つ**: 同じpersona_idは何度呼んでも整合した反応
- **経時更新**: 経済情勢・社会変化を反映して人格が動く
- **記憶**: 過去にAPI経由で投げかけられた質問への自分の回答を覚えている（一貫性担保）
- **コスト**: 常駐型は記憶管理・更新コストが高いが、競争優位の源泉

### 「精度の高いシミュレーション」だけが目的（Ryuzo明言）
- ペルソナごっこではなく**現実の人間集団の反応を高精度で再現**することが唯一のKPI
- 検証方法案:
  - 実際のサーベイ調査（同じ質問を本物のN=1000に投げる）と比較して**正答率測定**
  - Park論文の85%を超えることが Yobou版の科学的目標
  - 各ドメイン特化層は **その分野の実データ（購買履歴・転職実績等）** との一致率で評価

### 次論点（Ryuzo追加判断待ち）
1. **コア層1000人の実施コスト**: 2h × 1000人 × 単価いくら？外注 vs 自社 vs 学術連携
2. **常駐コスト**: N=10万-100万エージェントの記憶ストレージ・LLM更新サイクル
3. **競合Aaruの料金体系**を要調査（API単価・契約形態）
4. **データ提供パートナー**: ドメイン特化層を作るためのデータ持ち企業との提携設計

## 追記（2026-05-11 第3ラウンド：Ryuzoの「場面ごと最適生成」発想統合）

### Ryuzo追加発想
- **場面ごと最適エージェントを動的生成**するのもアリ
- ベース1000人があれば「こんな感じだよね」の情報源にできる
- 最初の1000人分を **Yobou自前で作る** べきか検討中

### 修正アーキテクチャ（コア常駐 × 動的派生 ハイブリッド）

```
Query（場面パラメータ）
  ↓
場面分析: domain/age/role/values抽出
  ↓
コア1000人から「該当層」k-近傍選定（常駐・状態あり）
  ↓
場面最適エージェントN派生（クエリ時生成・場面に最適化・状態なし）
  ↓
並列シミュレーション → 反応集約 → API応答
```

| 層 | 構造 | 役割 |
|---|---|---|
| **コア1000人** | 常駐・状態あり・深層インタビュー由来 | 全シミュの基盤データ「こんな感じだよね」の源泉 |
| **動的派生N人** | クエリ時生成・状態なし | 場面ごとに最適化された反応生成 |

→ コスト爆発（10万常駐の記憶管理）を回避しつつ、「場面最適」と「1000人ベース」両方を満たす。

### 1000人を Yobou 自前で作るか

| 選択肢 | 質 | コスト | 速度 | データ所有 | 差別化 |
|---|---|---|---|---|---|
| 完全自前 | ★★★ | 1000万級（2000時間） | 遅い | ◎完全所有 | ◎最強 |
| 外注（リクルート/インテージ） | ★★ | 500-800万 | 速い | △要交渉 | ○ |
| 学術連携（松尾研/東大） | ★★★ | 安 | 遅い | △公開縛り | △ |
| クラウドソース（CrowdWorks等） | ★ | 100-200万 | 速い | ◎ | △ |
| **Phase自前**（推奨） | ★★★→★★ | 段階分散 | 段階 | ◎ | ◎ |

### 推奨：Phase自前

- **Phase 1（5-7月）**: 自前100人 — Ryuzo人脈・X/SNSフォロワー・イベント参加者から募集。質確立・インタビュー手法ノウハウ化。MVP用にユニオンツール提案で使える。コスト数十万（謝礼2-3k×100+工数）
- **Phase 2（秋）**: 学術連携 or 外注で残り900人 — Phase 1の質基準を移植
- **Phase 3（年末-）**: 派生層自動生成 — API課金開始

### 自前にすべき決定的理由
**ペルソナデータ = Yobouの堀**。外注・公開すると競合がコピー可能。**Yobou自前データセットが競争優位の源泉**になる。

### Phase 1 = MVP100人モデルでユニオンツール提案
- N=100でも「うちのサイト訪問者のリアクション分布」をデモ可能
- 完璧なN=1000を待たずに**売上を立てて投資原資を作る**戦略
